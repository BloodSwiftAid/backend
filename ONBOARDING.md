# SwiftAid Backend - Architectural Design & Onboarding

Welcome to the SwiftAid Backend! This document serves as an architectural overview and onboarding guide for new developers joining the team.

## System Architecture

The SwiftAid backend is built as a RESTful API using **Django** and **Django REST Framework (DRF)**. It leverages **PostgreSQL** for relational data, **Redis** for asynchronous task queues (Django-RQ) and caching, and **Django Channels** for WebSocket support.

### Tech Stack
- **Language**: Python 3.12+
- **Framework**: Django 6.x & Django REST Framework
- **Database**: PostgreSQL (via `psycopg2`)
- **Caching & Queues**: Redis (via `django-rq`)
- **WebSockets**: Django Channels & Daphne
- **Authentication**: JWT (JSON Web Tokens) via `rest_framework_simplejwt`
- **Dependency Management**: `uv` (pyproject.toml)

### Core Apps & Modules
The system is divided into several loosely coupled Django apps:

1. **`core`**
   - Contains project-wide settings, base models (`BaseModel`), custom middleware, utility functions, error handlers, and pagination configurations.
   - Settings are modularized using `django-split-settings` inside `core/settings/`.

2. **`users`**
   - Handles authentication, authorization, and custom user models.
   - Roles include: `HOSPITAL_ADMIN`, `HOSPITAL_STAFF`, `BLOODBANK_ADMIN`, `BLOODBANK_STAFF`, `PUBLIC_USER`, `DISPATCH_RIDER`, and `INTERNAL_ADMIN`.
   - Manages Organizations (`Hospital`, `BloodBank`), user profiles, and OTP verification.

3. **`inventory`**
   - Manages blood bank inventories, blood group stocks, and storage capacities.

4. **`transaction`**
   - Handles the core business logic of blood requests, tracking orders between hospitals and blood banks.

5. **`payment`**
   - Handles financial transactions, wallets (`wallet_balance`), system commission logic, and payouts (e.g., Paystack integration).

6. **`notification`**
   - Responsible for sending emails, SMS, and real-time in-app notifications (WebSockets).

## Design Patterns & Conventions

### 1. Base Models
Always inherit from `core.model.BaseModel` when creating new models. It provides standard fields like `created_at` and `updated_at`.

### 2. Settings Management
We use `django-split-settings`. Settings are categorized into:
- `components/`: Logical chunks (e.g., `common.py`, `db.py`, `email.py`, `logger.py`).
- `environments/`: Environment-specific overrides (e.g., `dev.py`, `local.py`, `prod.py`).

### 3. Authentication & Permissions
All endpoints require authentication by default (configured in `REST_FRAMEWORK` settings).
- **Tokens**: Pass `Authorization: Bearer <token>` in the header.
- **Permissions**: Custom permissions exist in `users/permissions.py` (e.g., `IsFacilityVerified`). Always assign precise permissions to your views.

### 4. Background Tasks
We use `django-rq` and Redis for background processing (e.g., email sending, report generation).
- Avoid blocking the main thread.
- Queue tasks in `RQ_DEFAULT_QUEUE` or `RQ_SENDER_QUEUE`.

### 5. Media & File Uploads
Media is stored using Cloudinary (`django-cloudinary-storage`). File fields on models seamlessly integrate with it for images and KYC documents.

## First Steps for Onboarding
1. Read through the models in `users/models.py` to understand the domain relationships (Users, Hospitals, BloodBanks).
2. Familiarize yourself with `core/settings/components/common.py` for a high-level configuration overview.
3. Review the API documentation (Swagger/Redoc) generated by `drf-spectacular` by navigating to `/api/schema/swagger-ui/` when running locally.
